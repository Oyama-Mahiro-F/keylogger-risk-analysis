# 键盘监听风险机理分析与防护验证

> 信息安全编程技术与实例开发 · 平时作业

---

## 项目简介

一个围绕键盘监听（Keylogging）安全风险的编程作业，用 Python 写了三个核心模块：

1. **风险模拟** — 模拟键盘 Hook 的行为特征，搞清楚键盘记录器到底是怎么工作的
2. **检测识别** — 从文件、进程、注册表、API 四个维度扫描系统，给出风险评分
3. **防护验证** — 实现三种防键盘监听的输入方案，对比各自的效果

代码量约 1300 行，10 个 Python 文件，外加一个 29 页的 PDF 报告。

## 模块说明

### 风险模拟 (`risk_simulation/`)

#### key_hook.py

`KeyboardHookSimulator` 用 pynput 的 `Listener` 监听键盘事件，记录按键和时间戳。

几个有意思的细节：

- **隐蔽存储** — 日志写到 `~/.sys_cache/.idx.dat`，伪装成系统缓存目录
- **分块写入** — 不是每次按键都写盘，而是攒够 20 条才 flush 一次。真实恶意软件也这么干：减少 I/O 痕迹、模拟批量上报、降低 CPU 占用避免被用户察觉
- **线程安全** — `_on_press` 回调在键盘事件线程，`_flush_buffer` 可能被 timer 或主线程触发，用 `threading.Lock` 保护共享缓冲区
- **异常静默** — 写日志失败时 `except Exception: pass`，不抛异常。恶意软件的优先级是"不被发现"高于"功能完整"，崩溃弹框等于自曝

模块里还存了一份 `KNOWN_KEYLOGGER_SIGNATURES`，收录了 7 个键盘 Hook 相关 API（SetWindowsHookExA/W、GetAsyncKeyState、GetKeyState、GetKeyboardState、GetRawInputData、RegisterRawInputDevices），每个 API 的用途都有说明。

`simulate_hook_api_calls()` 模拟了一条恶意调用链——先轮询 Shift/Ctrl 状态键，再注册 WH_KEYBOARD_LL 钩子，然后持续读取按键。这是键盘记录器的标准"启动-运行"序列。

#### logger.py

`SmartKeyLogger` 在基础 Hook 上加了三层能力：

1. **窗口标题关联** — 检测到活动窗口切换时，在日志里插入 `[窗口切换: Chrome → 网银登录页]` 这样的标记。这是关键：攻击者关心的不是"你按了什么键"，而是"你在哪个应用里按的这些键"
2. **敏感模式匹配** — 6 种正则（密码字段、信用卡号、身份证号、手机号、邮箱、IP），对最近 500 字符的缓冲区做实时扫描。500 字符窗口是为了控性能，避免长文本下正则回溯爆炸
3. **定时快照** — 周期性保存按键缓冲区上下文，可以看到键盘记录随时间的变化

### 检测识别 (`detection/`)

#### hook_detector.py

`HookDetector` 从两个维度做检测：

**文件系统扫描** — 在 `~/.sys_cache`、`%TEMP%`、`%LOCALAPPDATA%/Temp` 三个路径下递归搜索，匹配 5 种可疑文件/目录命名模式（`.idx.dat`、`keylog.*.dat`、`.sys_cache` 目录等）。递归深度限制在 3 层，避免在 `%TEMP%` 这种可能有几万个子目录的地方跑太久。

**API 签名检测** — 用 ctypes 直接访问 user32.dll 和 kernel32.dll，检查 9 个 Hook 相关 API 是否在系统中可用。API 存在本身不代表被恶意使用（Windows 合法程序也大量用这些 API），所以这个维度的权重不高。

风险分按发现项的严重程度累加：高危 × 30、中危 × 15、低危 × 5。

#### process_scanner.py

`ProcessScanner` 用 psutil 遍历所有进程，每个进程从四个维度打分：

| 检测维度 | 触发条件 | 加分 |
|---------|---------|------|
| 进程名可疑 | 匹配 keylog/hook/spy/capture/logger/monitor | +20 |
| CPU 异常 | >50% | +5 |
| 文件句柄过多 | >100 个 | +10 |
| 可疑网络连接 | 目标 IP 命中黑名单 | +15 |

总分 ≥15 就标记为可疑。进程名匹配一项就能触发（+20），CPU 异常（+5）和句柄过多（+10）各一半加起来也是 15——这意味着 Chrome 渲染重页面、IDE 等正常场景也可能被标记，误报可以接受，最终判断需要人工确认。

psutil 不可用时会返回预置的模拟数据（冒充 svchost.exe 但在 `C:\Users\Public\` 下运行、冒充 winlogon.exe 但在 `C:\Windows\Temp\` 下运行），这两个例子参考了真实恶意软件的常见伪装手法。

#### risk_scorer.py

核心算法：**四维度加权评分**。

```
总分 = 文件系统分 × 0.35 + 进程行为分 × 0.30 + 持久化分 × 0.20 + API特征分 × 0.15
```

权重不是随便给的：

- **文件系统 35%** — 键盘记录器必然产生日志文件，这是最直接的物理证据，没法隐藏
- **进程行为 30%** — 运行时必然有特定的 API 调用和行为模式，区分能力较强
- **持久化 20%** — 不是所有键盘记录器都持久化（有些就跑一次），所以权重略低
- **API 特征 15%** — API 存在 ≠ 恶意使用，Windows 系统本身大量使用这些 API，权重大了会误报成片

每个维度内部有 4-5 个检测指标，各指标有预设分值。根据实际检测发现与指标的匹配程度（用严重程度加权：high=100%、medium=60%、low=30%）算出维度得分，归一化到 0-100 后再乘权重。

最后按总分落区间：
- 0-29：低风险（绿色）
- 30-59：中风险（黄色）
- 60-79：高风险（橙色）
- 80-100：严重风险（红色）

每个区间对应的处置建议也不一样——严重风险要求立即终止进程、隔离主机，低风险只需要保持常规监控。

### 防护验证 (`protection/`)

#### anti_hook.py

`AntiHookProtection` 模拟了一个反 Hook 防护引擎。核心是一个后台线程，每 2 秒检查一次是否有新的 Hook 注册。2 秒间隔的考量：太短（<500ms）耗电且抢 CPU，太长（>10s）攻击窗口太大。

`generate_protection_report()` 返回一个防护策略清单和 7 条最佳实践建议（安全桌面、UAC、审计自启动项、保持更新、不运行不明来源程序、双因素认证、生物识别）。

#### secure_input.py

`SecureInputEngine` 实现了三种互补的防护方案：

1. **虚拟键盘输入** — 按键事件不经过物理键盘通道，传统 Hook 抓不到。缺点是屏幕截图和鼠标 Hook 能绕过
2. **输入混淆** — 真实按键中随机掺入 40% 概率的噪声字符（随机字母、退格键、ESC），退格键删掉噪声保证最终输入正确。缺点是高级记录器可以分析退格序列还原
3. **加密通道输入** — 用 SHA256(当前纳秒时间) 派生一次性会话密钥，对每个按键做 XOR 加密后 Base64 编码。键盘记录器只能抓到密文。这是三种方案里安全性最高的

三种方案都返回 `InputProtectionResult` 数据结构，包含防护等级、性能开销、是否抗键盘记录等统一评估指标，可以横向对比。

每种方案还会跑一次"模拟键盘记录器尝试"——展示如果真的有键盘记录器在跑，它会抓到什么。虚拟键盘 → 空结果，输入混淆 → 乱码序列，加密通道 → 密文字符串。

## 一些实现上的选择

**为什么用 Python 而不是 C/C++？**

这个作业的重点是理解风险机理和写报告，不是做一个能实际对抗杀软的工具。Python 写起来快，库丰富，出报告也方便。1300 行 Python 可以把三个模块都讲清楚，同等工作量用 C 可能连 Hook 检测都没写完。

**为什么没装 pynput 也能跑？**

作业最终要交 PDF 报告给老师审阅，不能假设老师的电脑上装了 pynput 和 psutil。程序启动时检查依赖，缺什么就自动切到模拟模式——pynput 没了就展示 API 签名和调用链，psutil 没了就用预置的模拟数据。零配置就能跑，老师不需要折腾环境。

**为什么 PDF 报告用 Python 脚本出而不是 Word/LaTeX？**

fpdf2 是纯 Python 的，不依赖外部程序。老师拿到代码后跑 `python generate_report.py` 就能复现完全一样的 PDF，可以验证报告是代码自动产生的、内容来自运行数据而非手动排版。

**三层架构是不是过度设计了？**

考虑过把所有功能写在一个文件里（更简单），但三个阶段的输入输出都不一样，揉在一起反而不清楚。分开后每个模块可以独立测试，而且报告里"代码开发"那章按模块讲也更顺。
